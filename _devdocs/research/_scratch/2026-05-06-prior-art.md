# pmstate prior art survey

**Date:** 2026-05-06
**Status:** External validation pass for the "this is its own category" claim in the v1 design doc.
**Method:** Web survey of statecharts, durable workflows, DAG/dataflow tools, agent frameworks, event-sourcing libraries, filesystem-as-DB tools, PM data models, and recent (2025-2026) entrants.

---

## 1. TL;DR

- **Closest neighbor by mechanics: Apache Hamilton.** Its caching key (`node_name + code_version + dependencies_data_versions`) is structurally the same trick pmstate uses for parent rollup invalidation. Hamilton is dataflow, not process state, and has no agent surface â€” but the build-system-style content-hash recompute is the same idea applied differently. ([Hamilton caching docs](https://hamilton.apache.org/concepts/caching/))
- **Closest neighbor by feel: `langchain-ai/deepagents`.** A Claude-Code-inspired harness with a filesystem backend, planning tool, and `task` subagent dispatch. Same lineage as pmstate's "deep agent" but no hierarchical state model, no rollup, no per-leaf event log â€” it's a generic agent harness, not a process-state framework. ([deepagents repo](https://github.com/langchain-ai/deepagents))
- **Genuinely novel about pmstate:** the *combination* of (a) tree-of-nodes-as-process, (b) per-leaf append-only JSONL where rows ARE events, (c) build-system-style content-hash lazy rollup, (d) views-not-raw-logs as the structural context-bound, and (e) one-agent-as-observer over that tree. No surveyed tool combines more than two of these.
- **Borrow heavily from:** Hamilton's cache-key shape; LangGraph's `Checkpointer` interface contract; Backlog.md's "filesystem-is-the-product" framing; Pydantic-AI's `FileStatePersistence` ergonomic; Letta Filesystem's "files outscore specialized memory" benchmark as social proof; Claude Code's `CLAUDE.md`/Skills progressive-disclosure pattern.
- **Naming risk: low but non-zero.** No `pmstate` exists on PyPI/GitHub. Closest collisions: `pystates` (state-machine library), `pm-brain` (git-versioned PM templates). "pm-state" is essentially unclaimed. Worth grabbing the PyPI/npm names early.

## 2. Per-tool capsules

### Hierarchical state machines

**XState v5 (statelyai).** Hierarchical/parallel statecharts with first-class actor model; v5 introduces actor systems with implicit hierarchy and registry lookup via `systemId`. Persistence is supported via `actor.getPersistedSnapshot()` / `restore` for both client (page refresh) and server (workflow) use cases. No agent integration. Statecharts SCXML semantics. **Relevance:** vocabulary borrow (state actors, events-as-transitions, hierarchical compose) but XState is in-memory primary, disk is an export; pmstate is disk-primary. ([XState v5 announcement](https://stately.ai/blog/2023-12-01-xstate-v5))

**SCXML / robot.** SCXML is the W3C statecharts spec â€” heavyweight, XML-based, no Python ergonomics. `robot` (jamiebuilds) is a 1KB FSM library, no hierarchy. Neither relevant beyond academic vocabulary.

### Durable workflow engines

**Temporal.** The reference durable execution platform. Hierarchical via *child workflows* â€” but child workflows do NOT share state with parents; they communicate only via async signals, persisted as events in the workflow event history. This is structurally the *opposite* of pmstate's rollup model: Temporal isolates child state, pmstate aggregates child state. Heavy infra (cluster, server). **Relevance:** event-history-as-source-of-truth is the right shape; the "no shared state, only signals" pattern is the discipline pmstate explicitly *doesn't* enforce. ([Temporal child workflows](https://docs.temporal.io/child-workflows))

**Restate (newest, closest to events-as-substrate).** Lightweight sidecar; the server is essentially a durable log (Bifrost) plus processors. Events are the primary durability layer; state is materialized from events. Conceptually closer to pmstate's "events are leaf logs" insight than Temporal's separate event-history abstraction. No hierarchical state, no agent surface. **Relevance:** the principle "log first, state is a view of log" is the same; pmstate scopes that principle to a per-node JSONL file rather than a partitioned distributed log. ([Restate from-first-principles blog](https://www.restate.dev/blog/building-a-modern-durable-execution-engine-from-first-principles))

**DBOS Transact.** Postgres-backed durable workflows; `@DBOS.workflow()` and `@DBOS.step()` decorators checkpoint state in Postgres. As of March 2026, supports live-patching running workflows and Pydantic-AI integration that auto-checkpoints agent runs. **Relevance:** ergonomic target â€” minimal decoration, Postgres as substrate. pmstate trades Postgres for filesystem; the ergonomic shape (decorator-light, "your existing code is durable") is worth borrowing in spirit. ([DBOS workflows docs](https://docs.dbos.dev/python/tutorials/workflow-tutorial))

**Resonate.** Distinguishes itself from DBOS by load-balancing across Worker groups; multi-Worker orchestration. Less relevant â€” pmstate has one writer per leaf by design. ([Resonate vs DBOS](https://docs.resonatehq.io/evaluate/coming-from/dbos))

**Inngest.** Step-based memoization: each step runs once, result persisted, subsequent executions skip completed steps via stored results. Event-driven trigger model. Closer to durable functions than to hierarchical process state. **Relevance:** "step result is the cache, code is the function key" is structurally similar to Hamilton's content-hash; less directly applicable. ([Inngest principles](https://www.inngest.com/blog/principles-of-durable-execution))

### DAG / dataflow

**Apache Hamilton (most relevant tool surveyed).** Dataflow DAG defined as Python functions; the cache key is `(node_name, code_version, dependencies_data_versions)` where `code_version` is a hash of the function source ignoring docstrings/comments. Crucially: small `data_version` strings are passed through the dataflow rather than actual data, and "by traversing the cache keys' `dependencies_data_versions`, we can reconstruct the dataflow structure." Known limitation: code hash does not cover nested function calls. **Relevance: this IS pmstate's content-hash rollup mechanic, applied to dataflow rather than process state.** The cache-key shape is borrowable verbatim. ([Hamilton caching docs](https://hamilton.apache.org/concepts/caching/))

**Dagster.** Software-defined assets with hierarchical asset keys (a list of strings = filesystem-like namespace). Strong lineage UI. As of May 2026 charges per-materialization on Solo/Starter plans. **Relevance:** the asset-key-as-path convention validates pmstate's `/active/fieldwork/enum_log` addressing. Dagster is platform-shaped (a server, a UI, a billing model); pmstate is library-shaped. ([Dagster assets API](https://docs.dagster.io/api/dagster/assets))

**Prefect 3.** Python-native, dynamic, not strictly DAG-shaped. Flows can branch at runtime. **Relevance:** Prefect's "tasks shouldn't be a rigid DAG" stance aligns with pmstate's `spawn`/`prune` runtime tree mutation. ([Prefect comparison](https://www.prefect.io/compare/dagster))

**Airflow.** Schedule-based DAGs, batch ETL/ELT, mature but rigid. Not relevant beyond historical context.

### Agent frameworks with state

**LangGraph.** Built-in `Checkpointer` interface (`langgraph.checkpoint.base.BaseCheckpointSaver`) â€” implementations must provide `.put`, `.get_tuple(config_with_thread_id_and_checkpoint_id)`, `.list`, and `.get_next_version()`. State is keyed by `thread_id` and stores `channel_versions` + `channel_values` per checkpoint. Multiple backends: in-memory, SQLite, Postgres, Redis, Couchbase. Supports time travel and human-in-the-loop pausing. **Relevance: the strongest *interface contract* in the agent space.** pmstate could literally implement a `LangGraphCheckpointer` adapter that maps `(thread_id, checkpoint_id) -> JSONL row range` if interop matters. ([LangGraph checkpoints reference](https://reference.langchain.com/python/langgraph/checkpoints))

**`langchain-ai/deepagents` (closest agent-side cousin).** Claude-Code-inspired harness on LangGraph. Tools: `write_todos` (planning), `read_file`/`write_file`/`edit_file`/`ls`/`glob`/`grep` (filesystem), `execute` (shell), `task` (subagent with isolated context). Auto-summarizes long conversations, saves large outputs to files. Four-line minimum to working agent. 22.3k stars, MIT. **Relevance: this is what pmstate's harness layer probably looks like in spirit.** The `task` subagent pattern directly parallels pmstate's deferred-to-v0.2 `summarize_branch`. Different framing: deepagents is general-purpose; pmstate is over a structured tree. ([deepagents repo](https://github.com/langchain-ai/deepagents))

**Pydantic-AI graph.** `pydantic_graph` provides three persistence implementations: `SimpleStatePersistence` (in-memory latest), `FullStatePersistence` (in-memory list of snapshots), `FileStatePersistence` (JSON file). Authors subclass `BaseStatePersistence` for production. **Relevance: `FileStatePersistence` is the closest existing API to pmstate's `Log`/`Table` storage helpers â€” directly worth studying as an API ergonomic reference.** ([pydantic_graph.persistence](https://ai.pydantic.dev/api/pydantic_graph/persistence/)) Also: DBOS now auto-checkpoints Pydantic-AI agents to Postgres ([DBOS March 2026](https://www.dbos.dev/blog/dbos-new-features-march-2026)).

**Letta (formerly MemGPT).** OS-style memory hierarchy: in-context core memory + archival/recall memory. "Letta Filesystem" stores conversational histories in files and scored 74.0% on LoCoMo benchmark, beating specialized memory libraries. Letta is now transitioning Filesystem toward actual filesystem access + context repos. **Relevance: empirical validation that filesystem-as-memory wins for agents** â€” useful social-proof citation for pmstate's Principle 3. ([Letta benchmarking blog](https://www.letta.com/blog/benchmarking-ai-agent-memory))

**CrewAI / OpenAI Agents SDK / smolagents / Agno / AutoGen.** All are multi-agent orchestration frameworks. None have hierarchical *state* (they have hierarchical *agents*). Different category â€” pmstate is one observer over structured data; these are many actors over a shared task. Not direct neighbors. ([Lindy 2026 review](https://www.lindy.ai/blog/best-ai-agent-frameworks))

### Event-sourcing in Python

**`eventsourcing` (PyEventSourcing, v9.5.4 March 2026).** Mature, comprehensive ES library. Optimistic concurrency control (no two events with same sequence number). SQLAlchemy/Django backends; KurrentDB (formerly EventStoreDB) extension. **Relevance: it's the right design pattern but the wrong scale and substrate** â€” pmstate doesn't want a record/event class hierarchy with aggregate roots; it wants flat JSONL rows on disk. Useful as a reference for envelope shape and version-field discipline. ([eventsourcing on PyPI](https://pypi.org/project/eventsourcing/))

**CloudEvents.** CNCF spec for event envelopes; Python SDK at `cloudevents`. Defines `id`, `source`, `type`, `specversion`, `data` fields. **Relevance: pmstate's event envelope is structurally identical except for the `causation_id` extension.** Adopting CloudEvents JSON format would make events interop-friendly at zero cost. Recommended adoption â€” see Borrowable Patterns section. ([CloudEvents JSON format](https://github.com/cloudevents/spec/blob/main/cloudevents/formats/json-format.md))

### Filesystem-as-DB / inspectable

**Datasette.** SQLite-as-source-of-truth for *publishing* data; Datasette is the read tool. v1.0a28 in April 2026. Not write-side. **Relevance: validates the "inspectable file = product" stance** but pmstate is on JSONL not SQLite. ([Datasette release](https://simonwillison.net/2026/Apr/17/datasette/))

**Dolt.** Git-versioned SQL tables. Versions cells, not files. **Relevance: the "git for data" framing is adjacent but Dolt forces structured tables;** pmstate's JSONL events are git-friendly without ceremony.

**Datomic.** Immutable database, time-travel queries first-class. Closed source for years, recently free. EAV model is heavier than pmstate needs. **Relevance: time-travel-as-default validates the principle but the data model is too heavy.**

**dbt + DuckDB.** Files as inputs, derived assets as outputs. Not state-flow.

**Backlog.md (newest neighbor, PM-side).** CLI tool that turns a Git repo into a project board; tasks/drafts/decisions are markdown files with YAML frontmatter in `backlog/` directory. AGENTS.md instruction file for LLM agents, structured plain-text outputs. **Highly relevant prior art for the framing "filesystem IS the project board."** Different domain (kanban tasks vs hierarchical process state) but the README/docs framing is borrowable. ([Backlog.md repo](https://github.com/MrLesk/Backlog.md))

### PM-flavored data models

**Linear.** Issues with parent-child via "sub-issues"; Projects contain Issues; Cycles are time-windows. Flat-ish hierarchy.

**Plane (OSS Linear clone).** Workspace > Project > Epic > Work Item > Sub-task. Hierarchy view shows parent-child. Confirms a 4-5-level hierarchy is the typical PM ceiling â€” pmstate's recursive composability is unusual in this space. ([Plane work items](https://docs.plane.so/core-concepts/issues/overview))

**Notion page tree.** Arbitrary recursive blocks. Closest in shape to pmstate's recursive `Node`. But Notion is content, not state-machined process.

**Asana tasks.** Projects > Tasks > Subtasks. Flat by convention.

### 2025-2026 newcomers worth flagging

- **Letta v1 architecture** (2026): explicitly cites lessons from Claude Code in its agent-loop redesign. ([Letta v1 blog](https://www.letta.com/blog/letta-v1-agent))
- **deepagents** (LangChain, 22k+ stars): the strongest claim-stake on "Claude-Code-style harness as a library."
- **Backlog.md**: filesystem-PM framing.
- **DBOS + Pydantic-AI integration** (March 2026): auto-checkpointing agents.
- **The "files are all you need" debate** (early 2026, Jerry Liu / Anthropic Skills / Letta benchmarks): the conceptual zeitgeist pmstate is riding. ([New Stack analysis](https://thenewstack.io/ai-agent-memory-architecture/))

## 4. What is genuinely novel

Honest accounting of the v1 design's eight principles against prior art.

1. **"The cheap tools can't accidentally be expensive."** Structural context-bound at every read path via views-not-raw-logs. **No surveyed tool has this as a stated principle.** Claude Code's paginated `Read` is the closest behavior, but it's a tool implementation detail, not an architectural commitment. **Genuinely novel as a framework principle.**

2. **Generic defaults, opt-in smarts.** The "zero-author-code working tree" stance. Most frameworks (LangGraph, Hamilton, Temporal) require *some* author code to do *anything*. deepagents gets close (4-line working agent) but has no domain model. **Mostly novel.**

3. **Filesystem is the substrate.** Not novel in 2026 â€” Backlog.md, Letta Filesystem, Anthropic Skills, deepagents all do this. **The zeitgeist is here. pmstate is riding it, not inventing it.** That's fine â€” but the marketing claim should not be "filesystem-first." It should be "filesystem-first PROCESS STATE."

4. **Agent as observer, not actor.** Most agent frameworks (CrewAI, AutoGen, OpenAI Agents SDK, even deepagents) frame the agent as an actor. pmstate's deliberate constraint to *one* agent that *only reads* is unusual. **Genuinely novel positioning** in the agent-framework space.

5. **Lazy everything.** Hamilton has lazy recompute. LangGraph has lazy checkpoint loads. Most tools have *some* laziness. **The combination of lazy navigation + lazy rollup + lazy log reads as a unified principle is novel.**

6. **Linear growth.** "One new node = one new line." This is an ergonomic claim, not a mechanism. deepagents/Backlog.md hit similar marks for their domains. **Aspirational, not novel.**

7. **Library not platform.** Resonate, DBOS, Inngest, Temporal are all platforms. LangGraph, Hamilton, deepagents are libraries. **Common stance, not novel.** What IS slightly distinctive: pmstate's commitment to a *two-format surface* (`Log`, `Table`) and refusing to grow more. That kind of surface-area discipline is rare.

8. **Recursive composability over specialized types.** A single `Node` primitive at every level. Hamilton has this for nodes; XState has it for states. **Common pattern, not novel.**

**Net assessment:** the framework's most novel claims are #1 (structural context-bound via views), #4 (agent as observer over structured tree), and the *combination* embodied by the matrix. The "tree of nodes with per-leaf JSONL where rows ARE events with content-hash parent rollup readable by one observing agent" sentence is, as far as this survey goes, genuinely unoccupied.

What's NOT novel and shouldn't be claimed as such: filesystem-as-substrate, content-hash caching, hierarchical state, library-vs-platform.

## 5. Naming risk

**Direct collisions on `pmstate` / `pm-state` / `pm-state-framework`:** none found on PyPI or GitHub.

**Adjacent names worth knowing about:**
- `pystates` â€” Python state-machine library (egradman). Different name, different domain.
- `pm-brain` â€” git-versioned PM templates (andreaskelm). Same prefix, different shape.
- `pysm` (pgularski) â€” Python state machine. Phonetically close-ish.
- `statesman` (opsani), `transitions` (pytransitions), `state_machine` (jtushman) â€” generic FSM libraries.

**Recommendation:** name is essentially unclaimed. Worth registering `pmstate` on PyPI (and the GitHub org/repo) early to prevent squatting. The "pm-state" hyphenated form is also free.

**Marketing-name confusion risk:** "pmstate" reads as "PM state" (project-management state) and "process-machine state" simultaneously â€” both fit. No major overloaded acronym (unlike, say, `psm` which collides with power-system-management, parametric-state-machine, etc.).

## 6. Borrowable patterns

Concrete things to crib, with sources.

1. **Cache-key shape (Hamilton).** Use `(node_path, view_fn_code_hash, children_view_hashes)` as the rollup invalidation key. Pass small `view_hash` strings up the tree, not actual views, when computing whether to recompute â€” exactly the Hamilton "small `data_version` strings are passed through the dataflow" pattern. ([Hamilton caching](https://hamilton.apache.org/concepts/caching/))

2. **Be explicit about the nested-call hash gotcha.** Hamilton documents that "the cache will find a `cache_key` match and return" stale results when nested utility functions change. pmstate will have the same issue with view functions that call helpers. **Document this up front rather than discovering it via bug reports.** ([Hamilton caching](https://hamilton.apache.org/concepts/caching/))

3. **Checkpointer interface contract (LangGraph).** If pmstate ever wants to expose the tree as a LangGraph checkpointer (so users on LangGraph can plug pmstate in), implement `BaseCheckpointSaver` with `.put`, `.get_tuple`, `.list`, `.get_next_version()`. Map `(thread_id, checkpoint_id)` to `(node_path, jsonl_byte_offset)`. Cheap interop bridge, large social surface area. ([LangGraph checkpoints](https://reference.langchain.com/python/langgraph/checkpoints))

4. **`FileStatePersistence` API ergonomic (Pydantic-AI).** Steal the `BaseStatePersistence` subclass shape â€” three methods, file-backed, dirt simple. pmstate's `Log` and `Table` should look like this. ([pydantic_graph.persistence](https://ai.pydantic.dev/api/pydantic_graph/persistence/))

5. **CloudEvents envelope (CNCF).** Adopt CloudEvents JSON format for the leaf event row â€” `specversion`, `id`, `source`, `type`, `time`, `data`. Add `causation_id` as an extension attribute. Free interop with the entire CloudEvents ecosystem at zero cost. ([CloudEvents JSON format](https://github.com/cloudevents/spec/blob/main/cloudevents/formats/json-format.md))

6. **Backlog.md framing: "filesystem IS the project board."** The README pattern of "this directory is the product, the CLI is just an accessor" is exactly the framing pmstate's README needs. Borrow the directory-layout-first explanation rather than starting with API. ([Backlog.md repo](https://github.com/MrLesk/Backlog.md))

7. **deepagents' four-tool surface (planning + filesystem + execute + task).** pmstate's planned surface (`list_tree`, `get_state`, `find_state`, `read_log`) maps cleanly. The naming convention "verb_object" is consistent and worth keeping. Consider adding `summarize_branch` (deferred to v0.2) as the analog of `task`. ([deepagents repo](https://github.com/langchain-ai/deepagents))

8. **Letta Filesystem benchmark as social proof (LoCoMo 74.0%).** "Files outperform specialized memory libraries" is now a citable empirical result. Use it to defend the filesystem-substrate principle in the README without sounding like a manifesto. ([Letta benchmarking](https://www.letta.com/blog/benchmarking-ai-agent-memory))

9. **Dagster's hierarchical asset keys (`["active", "fieldwork", "enum_log"]`).** Validates pmstate's slash-path addressing as a path *list* internally; render with slashes for humans. Useful for doc clarity. ([Dagster assets API](https://docs.dagster.io/api/dagster/assets))

10. **DBOS's "patch a running workflow" framing.** As pmstate evolves, the question of "what happens when a node's view function changes mid-flight?" becomes the same problem. DBOS's March 2026 patching feature is worth studying for vocabulary. ([DBOS March 2026](https://www.dbos.dev/blog/dbos-new-features-march-2026))

## 7. Open questions for follow-up

1. **CloudEvents adoption â€” yes or no?** Adopting `specversion`/`id`/`source`/`type`/`time`/`data` is essentially free and gives interop. The cost is a slightly more verbose envelope. Worth a 30-min spike to write a CloudEvents-shaped event and see if it feels heavy.

2. **LangGraph checkpointer adapter â€” worth the maintenance?** A `pmstate.LangGraphCheckpointer` adapter would put pmstate trees into the LangGraph ecosystem (large surface) but pmstate's per-leaf model doesn't cleanly match LangGraph's per-thread model. May be more confusion than help. Defer to v0.2 unless a concrete user asks.

3. **Hamilton-style nested-call hash trap â€” how loud do we make the warning?** This is going to bite a real user eventually. README mention? Docstring on `Reducer`? Runtime warning when a reducer's source includes function calls? No clear right answer.

4. **Letta's transition from "Letta Filesystem" to "actual filesystem access."** They're moving toward direct fs ops + context repos. Worth tracking â€” if Letta v2 (or v3) lands on something close to pmstate's shape from the agent-memory direction, that's either a competitor or an integration target.

5. **Backlog.md's `AGENTS.md` convention.** Should pmstate adopt an `AGENTS.md` file at the tree root that the deep agent reads first? Mirrors `CLAUDE.md` in Claude Code. Cheap convention, possibly high signal.

6. **The "no plugin system" v0.1 decision.** Hamilton, Dagster, Prefect all have plugin ecosystems. pmstate's library-not-platform stance forecloses this â€” fine for v0.1, but the second-user pressure is going to push back. Worth thinking ahead about what "compose with pmstate" looks like vs "plug into pmstate."

7. **One-writer-per-leaf assumption.** The internal design doc flags this as the load-bearing assumption. None of the surveyed tools (except XState's actor model and Restate's per-partition processors) handle concurrent writers cleanly via filesystem â€” they all use a real log service. Worth rehearsing the failure mode before v0.2.
